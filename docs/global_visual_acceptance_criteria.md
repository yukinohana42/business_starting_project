# Global Visual Acceptance Criteria

## Purpose

この基準は、Day1からDay5、親向け説明資料、カード、ワークシート、4:5版、A4印刷物の共通合格基準です。

「読みやすい」だけでは合格ではありません。発見ゲーム型の学習体験として機能していることが必要です。

## 1. Child-Facing Infographics and Slides

各ページごとに以下を確認します。

| 項目 | 合格 | 不合格 |
|---|---|---|
| キャラクター | 具体的な行動をしている | 丸アイコン、顔だけ、立っているだけ |
| ゲーム要素 | 学習行動と結びついている | タイトルにゲーム用語があるだけ |
| 変化 | 相手のBefore/Afterが見える | 概念説明だけ |
| 子供の行動 | 見る、選ぶ、書く、話すが明確 | 読むだけ |
| 画面 | Mission / Case / Card / Mapとして見える | 見出し、本文、カードだけ |
| 安全性 | 勝敗、採点、報酬化がない | 点数、ランキング、ご褒美感がある |
| 可読性 | Zoom/印刷で読める | 小さすぎる、詰め込みすぎ |

## 2. Parent-Facing Materials

| 項目 | 合格 | 不合格 |
|---|---|---|
| 親の役割 | ファシリテーターとして描く | 先生、採点者、ゲームマスターとして描く |
| 声かけ | 問い、観察、伴走が中心 | 正解提示、評価、説教が中心 |
| 褒め方 | 行動プロセスを褒める | 結果、すごさ、儲かりそうを褒める |
| 安全性 | 起業を押し付けない | 将来不安や競争で煽る |
| デザイン | 落ち着いたが温かい | 無機質すぎる、子供向けゲームUIすぎる |
| 実用性 | そのまま進行に使える | 理念だけで具体的行動がない |

## 3. Cards

カード教材は、次の条件を満たします。

- 1カード1行動。
- 子供が選ぶ、並べる、書き込む、比較する、話すために使える。
- カードが単なるイラストや飾りではない。
- カード名が行動を示している。
- 正解カードではなく、問いや観察の入口である。

推奨カード:

- 観察カード
- 困りごと発見カード
- 価値変換カード
- 顧客カード
- 実験カード
- 反応メモカード
- ピッチ問いカード

## 4. Worksheets

ワークシートは、次の条件を満たします。

- 子供が自分の言葉で書ける。
- 正解欄ではなく観察欄になっている。
- 親が採点する形式になっていない。
- Before/After、誰が、どう変わる、次に何を試す、が書ける。
- 最後にFinal Missionを書く欄がある。
- 書きやすく、余白が十分ある。

## 5. Production Stop Conditions

以下のいずれかに該当する場合は、HTML/CSSやPNG生成へ進みません。

- キャラクターの行動がページごとに定義されていない。
- ゲーム要素がページごとに定義されていない。
- 子供が何をする画面なのかが定義されていない。
- 親向け資料で親が採点者になっている。
- ゲーミフィケーションが報酬や勝敗に見える。
- 単なる説明スライドになっている。
- 固定方針との整合チェック表がない。

## 6. Review Required Questions

制作物レビューでは、必ず次を問います。

- これは説明資料か、発見ゲーム型教材画面か。
- キャラクターは内容を動かしているか。
- ゲーム要素は子供の行動を進めているか。
- 画面を見た子供が次に何をするかわかるか。
- 親が評価者にならない設計か。
- 可読性のために体験設計を捨てていないか。

## 7. Image-First Acceptance Criteria

画像生成前提の教材では、次も確認します。

| 項目 | 合格 | 不合格 |
|---|---|---|
| 画像prompt | page type、character action、game element function、learner action、visible changeがある | 雰囲気や画風だけを指定している |
| キャラクター一貫性 | anchor promptとcharacter promptが使われている | ページごとに別人に見える |
| 文字処理 | 正確な日本語本文は後編集で載せる設計 | 画像生成に長文日本語を任せる |
| 画面の主役 | 場面、行動、手元、小物、変化が主役 | 見出しと説明カードが主役 |
| HTML/CSSの役割 | 文字載せ、整形、書き出し補助 | 全ビジュアルを直描きで完結させる |
| 生成後レビュー | candidateを視覚確認し、修正/採用理由を記録している | 最初の生成物をそのままfinal扱いしている |
| 小物の意味 | 虫眼鏡、ペン、ノート、カードが手元や学習行動と自然につながる | 小物が浮く、増殖する、顔や体に不自然に重なる |

大量生成へ進む前に、1ページpilotを生成し、人間が以下を確認します。

- これは発見ゲーム型の教材画面に見えるか。
- キャラクターは学習行動を担っているか。
- ゲーム要素は飾りではなく機能しているか。
- 既存IP、金儲け煽り、勝敗、採点に見えないか。

final採用前には、`PASS_FINAL`、`PASS_WITH_MINOR_FIX`、`REGENERATE_WITH_REVISED_PROMPT`、`REJECT` のいずれかで判定します。

## 8. Manual Handoff Acceptance Criteria

すべての教材で、画像は最初に `candidate` として扱います。ユーザーがGPT画像生成で作った画像をrepoへ戻した後、Codex reviewと人間承認を経てからでなければ `final` 扱いにしません。

| 項目 | 合格 | 不合格 |
|---|---|---|
| prompt packet | 最終prompt、チェックリスト、修正prompt、保存先、報告テンプレがある | promptだけがあり、戻し方やレビュー条件がない |
| 役割分担 | Codex、GPT画像生成、ユーザーの役割が明記されている | Codexが画像生成まで担当するように読める |
| API依存 | APIがなくても手動handoffで進められる | APIキーがないと作業停止になる |
| 視覚確認 | 画像を確認できない場合はhuman visual review requiredと書く | 見ていない画像をレビュー済みにする |

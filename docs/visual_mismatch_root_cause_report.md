# Visual Mismatch Root Cause Report

## Purpose

Day1 16:9 v2 PNGは、読みやすい説明スライドとしては成立していました。しかし、ユーザーが求めていた「ゲーミフィケーション」「同世代オリジナルキャラクター」「マンガ的1コマ」「ミッション進行」「発見ゲーム感」には届きませんでした。

この文書は、Day1だけでなくDay2以降、親向け資料、カード、ワークシート、バッジ、クエストマップ、今後の生成プロンプトで同じズレを再発させないための原因分析です。

## Summary

主因は、思想の不足ではなく、思想を画面・カード・ワークに変換するための合格基準が弱かったことです。

- `SOURCE_OF_TRUTH.md` は正しい。
- しかし `DESIGN.md` や固定情報源は、「キャラクターが何をしているか」「ゲーム要素がどの学習行動を進めているか」まで要求しきれていなかった。
- 前回の実装指示は、既存HTML/CSS維持、Zoom可読性、崩れ防止を優先し、発見ゲーム型の体験設計を合格条件にしなかった。
- レビューも「重なりがない」「読みやすい」を中心に見ており、「ユーザーが求めた体験に見えるか」を検出できなかった。

## 1. Fixed Policy Issues

### SOURCE_OF_TRUTH

`SOURCE_OF_TRUTH.md` の思想は大きく変える必要はありません。起業を押し付けない、価値は相手の変化、お金は価値が届いたサイン、ゲーミフィケーションは学びの足場という根本は正しいです。

問題は、Source of Truthが思想レベルで止まっており、視覚教材としての必須構造を指定していなかったことです。

### Design Principles

`DESIGN.md` と固定情報源の `03_DESIGN_PRINCIPLES.md` は、温かい、見やすい、1ページ1メッセージ、短い問い、人物の変化を重視していました。このため、実装者は「きれいで読みやすい説明資料」を作れば合格だと判断しやすい状態でした。

不足していた粒度:

- 子供向け資料は、Mission Board / Case File / Card Tableなどの画面として設計すること。
- 見出し、本文、カードを並べただけでは不合格であること。
- キャラクターは顔や名前ではなく、メモする、指差す、カードを並べる、反応を見るなどの行動をしていること。
- ゲーム要素はタイトル語ではなく、子供の「見る、選ぶ、書く、話す」を進めるUIとして機能すること。

### Gamification Design

`docs/gamification_design.md` は、ミッション、カード、バッジ、スタンプ、進捗マップを挙げています。しかし「言葉として置くだけでは不可」と明記していませんでした。

その結果、Page 1に「今日のミッション」、Page 5に「ビジネス探偵ミッション」、Page 6に「Final Mission」と書けば、ゲーミフィケーションを反映したように見えてしまいました。

### Character and Illustration Guidelines

既存方針では、キャラクターの役割や服装、表情は書かれていました。しかし、HTML/CSSで作る場合に「丸アイコンだけでは不可」と明記されていませんでした。

「簡易人物表現」という言葉が逃げ道になり、実装では頭部の丸、顔、名前ラベルが中心になりました。これはキャラクターではなく、人物記号です。

### Visual Storytelling Guidelines

マンガ的1コマ、手元、表情、ポーズ、Before/Afterの方針はありました。ただし、実装レベルの条件が弱く、「カードを並べる場面」ではなく「カードが並んでいる画面」でも合格できる状態でした。

### Session Gamification Map

各回のミッション、カード、Final Missionは整理されていました。しかし、それを画面UI、カード教材、ワークシートの具体構造へ変換する粒度が不足していました。

### Reward and Badge Guidelines

報酬化しない方針は明確でした。一方で、Discovery Stamp、Learning Badge、Reflection Stampを「どの学習行動の記録として画面上に置くか」が不足していました。

### Parent Materials

親向け資料は「落ち着いた説明」とされていましたが、落ち着きすぎて普通の文章資料になるリスクがありました。親向けにも、Facilitator Guide、Question Card、OK/NG Response Sheet、Observation Logなどの実用的な形が必要です。

### Existing HTML/CSS Reuse

「既存HTML/CSS構造を活かす」という指示は、破壊的変更を避ける意味では安全でした。しかし、v1が説明スライド構造だったため、v2もその骨格に引き戻されました。

## 2. Prompt Issues

前回のCodex指示は、以下を強く求めていました。

- 16:9 Zoom共有を最優先
- 1ページ1メッセージ
- 本文3から5行
- 既存v1を壊さない
- HTML/CSSの簡易人物表現
- PNG生成前の表示崩れ確認

これらは必要ですが、優先順位が高すぎました。結果として、「読みやすい」「崩れない」「既存構造に収まる」が合格の中心になりました。

弱かった条件:

- キャラクターがページごとに具体的な行動をしているか。
- ミッションカード、探偵ノート、Final Missionが子供の学習行動を進めているか。
- 画面が Mission Board / Case File / Card Table として見えるか。
- Page 4が「ユウとソラがカードを作る場面」になっているか。
- Page 5が「ビジネス探偵として観察する場面」になっているか。
- PNG生成前に、発見ゲーム型体験として不合格なら止めること。

## 3. Implementation Issues

### Character Became Round Icons

HTML/CSSで短時間に安全に作れる表現として、丸い頭、顔、体の簡易図形が選ばれました。これは実装安全性は高い一方、ポーズ、手元、道具、視線、行動がほぼ表現できません。

今後は、HTML/CSSで作る場合でも、inline SVG、手描き風SVG、既存オリジナル素材、画像生成素材のどれを使うかを先に決める必要があります。

### Character Roles Did Not Drive the Page

ユウ、ソラ、ミホの名前は入りましたが、行動が弱く、ページの意味を動かしていません。

- ユウは観察していない。横に置かれている。
- ソラはカードを作っていない。顔アイコンとしているだけ。
- ミホは問いを出すカードになっているだけ。

### Game Elements Became Labels

探偵ノート、ミッションカード、Final Missionは登場しましたが、子供が「見る、選ぶ、書く、話す」ためのUIとしては弱い状態でした。

### Page 4

Page 4は「ユウとソラが価値変換カードを並べる場面」ではなく、「3枚のカードが置かれた説明ページ」になりました。手、机、付箋、未完成カード、相談の吹き出しが不足していました。

### Page 5

Page 5は3例に絞ったことで読みやすくなりましたが、「ビジネス探偵ミッション」というより普通の例カードになりました。ケースファイル、証拠カード、観察欄、発見スタンプが不足していました。

### Missing Progress and Discovery UI

進捗ゲージ、クエストマップ、発見スタンプ、ケースファイル、学びバッジが弱く、学習体験の進行が見えませんでした。

## 4. Review Issues

`session_01_infographic_16x9_v2_render_review.md` は、表示崩れ、文字量、重なり、Zoom可読性をよく確認しました。しかし、ユーザー意図との一致を評価する基準がありませんでした。

見逃した点:

- キャラクターが行動しているか。
- 丸アイコンだけになっていないか。
- ゲーム要素が画面上で機能しているか。
- 発見ゲーム型の教材画面に見えるか。
- 子供が画面を見て何をするかが明確か。
- 普通の説明スライドに戻っていないか。

今後のレビューでは、可読性だけでなく、学習体験としての機能を同じ重さで評価します。

## 5. Impact on Future Materials

同じ方針とプロンプトのまま進むと、以下でも同じ失敗が起きます。

| 対象 | 再発リスク |
|---|---|
| Day2 Problem Hunter | 困りごとリストの説明スライドになり、観察マップや質問カードとして機能しない |
| Day3 Idea Builder | 顧客、問題、解決案の説明表になり、カードテーブルやラフスケッチの場面にならない |
| Day4 Experiment Designer | PMFの概念説明になり、実験ラボ、信号カード、反応ログとして動かない |
| Day5 Pitch Challenger | ピッチ構成の説明になり、発表、聞き手メモ、改善選択が見えない |
| 親向け説明 | 理念説明に寄り、Question CardやOK/NG Response Sheetにならない |
| カード | 切れる説明カードになり、選ぶ、並べる、比較する道具にならない |
| ワークシート | 正解欄つきプリントになり、観察ログや実験ログにならない |
| バッジ/スタンプ | 飾りか報酬に見え、学びの行動記録にならない |
| クエストマップ | ただの進行表になり、現在地と次の行動を示さない |
| 4:5版 | 説明スライドを縦積みにしただけになる |
| A4印刷物 | 文章資料になり、手を動かす教材にならない |

## Required Correction

次回からは、HTML/CSS/PNG生成の前に、ページごとに次の4点を定義します。

1. キャラクターが何をしているか。
2. ゲーム要素がどの学習行動を進めているか。
3. 子供または親がその画面で何をするか。
4. 相手のBefore/Afterまたは学びの変化がどこに見えるか。

この定義がない場合、生成へ進んではいけません。
